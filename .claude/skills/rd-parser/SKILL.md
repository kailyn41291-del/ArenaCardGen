# RD Parser — Arena Card Generator

> **🚨 警告**:`parse_song_list_from_text` 跟 `clean()` 內部的舞台關鍵字黑名單,**每一條都是被某份真實 RD 噴過才加進去的**。修改前先讀 [.claude/agents/rd-format-tester.md](../../agents/rd-format-tester.md) 跑一遍既有行為,**證明你不會破壞任何既有 case**。

## 兩個 parser 的分工

[main.py](../../main.py) 裡有兩個解析器,**用途不同,不要搞混**:

| 函式 | 行 | 輸入 | 輸出 | 用途 |
|---|---|---|---|---|
| `parse_song_list_from_text` | 186-334 | 雜亂 RD 純文字(可能含舞台說明、時間、空格) | 乾淨歌單純文字(每行一個項目) | 給 ParserWindow 子視窗用 |
| `parse_setlist` | 158-183 | 乾淨歌單純文字 | `[{type, lines}]` 字典列表 | 給主視窗用,結構化成 UI 可顯示的字卡 |

正常使用流程:`parse_song_list_from_text` 跑完 → 結果文字匯入主視窗 → `parse_setlist` 再跑一次 → 變成字卡。

## `parse_song_list_from_text` 三層 fallback 決策樹

```
輸入 = 多行純文字
   ↓
┌──────────────────────────────────────────────────────┐
│ Pass 1:逐行嘗試結構化辨識                             │
│   每一行 try 以下 pattern(任一命中即記錄並 continue): │
│                                                      │
│   1. SONG \d+ : <title>      → ('song', num, title)  │
│   2. S\d+ [: . ：] <title>   → ('song', num, title)  │
│   3. Talking \d+             → ('talking', None, ..)  │
│   4. TALKING-\d+             → ('talking', None, ..)  │
│   5. ^[轉转][場场] ...       → ('transition', ..)     │
│   6. 行內 \bVCR\b            → ('transition', VCR ..) │
│   7. Chaser[~~]?(.*)         → ('chaser', ..)         │
│   (第 1-2 條同時觸發 clean() 處理 title)             │
└──────────────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────────────┐
│ Pass 1 結果有 'song' → 結構化輸出,END               │
│   (S01 皮卡丘 / TALKING-1 / 轉場_X / Chaser~X)        │
└──────────────────────────────────────────────────────┘
   ↓ (Pass 1 沒抓到任何 SONG)
┌──────────────────────────────────────────────────────┐
│ Pass 2:檢測「兩欄格式」                              │
│   條件:                                              │
│     - >50% 的非空行有 \t / 全形空格 / 2+ 空格        │
│     - 第一欄長度 ≤10 字元(編號)                     │
│   如果是兩欄:                                        │
│     - 跳過標題行(含「歌曲/內容/歌名/曲目/歌手」)    │
│     - 跳過 "(註解)" 行                                │
│     - 取第二欄當歌名,clean(),去重                    │
│     - 自動編號 S01..N                                 │
└──────────────────────────────────────────────────────┘
   ↓ (不是兩欄)
┌──────────────────────────────────────────────────────┐
│ Pass 3:每行就是歌名                                   │
│     - 跳過 "(註解)" 行                                │
│     - clean()                                         │
│     - 長度 1-60 之間才收                              │
│     - 去重                                            │
│     - 自動編號 S01..N                                 │
└──────────────────────────────────────────────────────┘
```

關鍵要點:
- **Pass 1 命中任何 SONG → 後面 Pass 2/3 不跑**(連帶意思:雜亂格式裡只要有一個 SONG 編號,其他不被 SONG 編號覆蓋的行就丟失)
- **去重以 `seen_titles`(Pass 1)、`seen`(Pass 2/3) set 比對**,憑相同 title 去重
- **編號**:Pass 1 保留原 SONG 數字、Pass 2/3 重新自動編號

## `clean()` 函式做的事(順序很重要)

定義在 [main.py:189-208](../../main.py:189),**對每個歌名 title 字串套用**。**順序敏感** — 不要任意調整順序:

1. **剝括號內容**:`(中文版)` / `（中文版）` 都剝掉(全形與半形)
2. **剝 HTML tag**:`<font>` / `<br/>` 之類
3. **剝時間標記**:`3'45"` / `3'45/` / `3′45″` 等格式
4. **剝舞台說明**(下面有完整黑名單)
5. **壓縮多空格**為單空格
6. **strip 裝飾符號**:`■★▶▷◆◇→←·。,、//　'"–—–` 加空白

### 舞台關鍵字黑名單(會剝掉「關鍵字之後到行尾」的內容)

定義在 [main.py:196-201](../../main.py:196):

```
舞者 服裝 服装 [\w]*道具 小舞台 大舞台 升降 左台 右台
合音 合唱 Bass GT gt CO\d 鋼琴 鍵盤 钢琴 键盘
花朵 彩虹 彩條 氣球 風扇 风扇 噴桶 搖滾 摇滚
現身 退場 上場 下場 特製 特制 男舞 女舞 \*\d
樂手 配乐 竖琴 Disco Intro 打造 人生
```

**觸發條件**:關鍵字前面要有「分隔符」(空格、全形空格、頓號、逗號、`–`、`—`、`-`)。

**意思**:`"S1: 皮卡丘 服裝紅色"` 會被剝成 `"S1: 皮卡丘"`,但 `"S1: 服裝皮卡丘"`(關鍵字在標題開頭、沒分隔符)不會被剝。

> **加新關鍵字的規則**:
> 1. 必須是被某份**真實的 RD** 噴過才加,不要憑空加
> 2. 加之前先在 `rd-format-tester` agent 的「補充案例」區寫一個**會破的測試**
> 3. 加完關鍵字,該測試要從失敗變通過
> 4. 重跑既有所有測試,**全部仍要通過**

### 不會被 clean 處理的 case(刻意保留)

- `'`、`"`、`「」`、`【】` 等引號 — clean **不剝引號內容**(歌名常含引號)
- `feat.` / `ft.` — 不剝(由 `auto_split` 處理折行,不是 clean)
- 數字 — 不剝(`Chaser 2025` 是 valid)

## 各類項目的 regex 與輸出規則

### SONG 編號

| 輸入 pattern | 範例 | 輸出 |
|---|---|---|
| `SONG\s*(\d+)\s*[:：]\s*(.+)` | `SONG 1: 皮卡丘`、`SONG1：皮卡丘` | `S01 皮卡丘` |
| `\bS(\d+)\s*[:：\.]\s*(.+)` | `S1. 皮卡丘`、`S1：皮卡丘` | `S01 皮卡丘` |

編號保留原數字,zero-pad 到 2 位:`SONG 5: X` → `S05 X`、`SONG 23: X` → `S23 X`、`SONG 100: X` → `S100 X`(不截斷)。

**title 走 clean()**。

### TALKING

| 輸入 pattern | 範例 | 輸出 |
|---|---|---|
| `Talking\s*(\d+)`(不分大小寫) | `Talking 1`、`talking 03` | `TALKING-{n}` |
| `^TALKING\s*[-\s]*\d+` | `TALKING-2`、`TALKING 3` | `TALKING-{n}` |
| 其他 TALKING 沒有數字 | `TALKING` | `TALKING` |

### 轉場(繁簡都支援,輸出統一繁體)

| 輸入 | 輸出 |
|---|---|
| `轉場03` | `轉場_03` |
| `转场03` | `轉場_03` |
| `轉場VCR 02` | `轉場_VCR 02` |
| `轉場_娜娜KTV` | `轉場_娜娜KTV` |
| `轉場_娜娜KTV - 互動橋段` | `轉場_娜娜KTV`(剝 `-` 後說明) |
| `轉場` 純詞 | `轉場` |

### VCR(行內,非「轉場 VCR」)

| 輸入 | 輸出 |
|---|---|
| `Opening VCR` | `VCR Opening` |
| `Ending VCR` | `VCR Ending` |
| `VCR 1` | `VCR 1` |

注意:行內 VCR pattern 比較鬆,可能會誤抓 — 改 regex 之前先測。

### Chaser

| 輸入 | 輸出 |
|---|---|
| `Chaser~阿明` | `Chaser~阿明` |
| `Chaser ~ 阿明` | `Chaser~阿明` |
| `Chaser` 純詞 | `Chaser` |

`Chaser` 後面的內容也走 clean()。

## 已知邊界情況

按發生頻率排:

### 高頻

1. **舞台說明黏在歌名後** — 已用 30+ 關鍵字黑名單覆蓋,但每場活動都可能撞新關鍵字
2. **時間標記不同寫法** — `3'45"` / `3'45/` / `3′45″`(unicode 引號)— 已涵蓋
3. **裝飾符號** — `■` / `★` / `▶` / `→` 開頭或結尾 — 已涵蓋
4. **括號內附註** — `(中文版)` / `（純樂器版）` — 已剝
5. **同一歌出現兩次** — 已 `seen_titles` 去重(以 title 為 key)

### 中頻

6. **編號跳號** — `SONG 5` 然後 `SONG 8`,中間沒 `SONG 6, 7` — 保留原編號(輸出 `S05` 跟 `S08`)
7. **編號順序錯亂** — `SONG 3` 在前 `SONG 1` 在後 — 按出現順序輸出(不重新排序)
8. **兩欄格式但分隔符混用** — 同份 RD 有的用 tab、有的用 2 空格 — Pass 2 用 OR 條件,通常 OK
9. **繁簡混雜** — Parser 不做繁簡轉換,原樣保留(轉換在 ParserWindow UI 層,辨識後手動切繁/簡)

### 低頻 / 已知會出錯

10. **雙語標題**(中英並列):`S1: 皮卡丘 Pikachu` — 不會被破壞,但 clean() 不會合併語言
11. **emoji 在歌名內**:`S1: 皮卡丘🌸` — 目前不破不錯,但裝飾符號 strip list 沒涵蓋 emoji
12. **單引號歌名**:`Don't Stop Me Now` — 沒問題,clean 不會剝
13. **超長標題**:>60 字元 — Pass 3 會丟掉(`1 <= len(title) <= 60` 條件),但 Pass 1/2 不會
14. **空白行 + 全空白行** — 已 `strip()` + `if not ls: continue` 處理

### 已知會錯的 case(尚未處理,記錄留)

- `clean()` 用「關鍵字 + `\*\d`」當分隔(例如 `*1`、`*2` 是舞台位置標記),但如果歌名本身含 `*`(`Mr.*`)會誤切。實務上沒撞到。
- `轉場_VCR` 跟「轉場 然後 VCR」的差異:目前判斷規則是「轉場後緊接 VCR 視為複合」。如果 RD 寫 `轉場 (3 分鐘) VCR`,會解析成「轉場」+「(3 分鐘) VCR」兩列?目前 Pass 1 第 5 條捕獲整行為 transition,丟給 clean。沒有測試覆蓋。

## `parse_setlist` 的差異

[main.py:158-183](../../main.py:158)。比 `parse_song_list_from_text` 簡單很多:

- 預期輸入是「乾淨歌單」(已經跑過 parser 一次),所以**不做 clean()**
- 對每行套 regex 判斷類型,直接結構化
- `song` 類型會呼叫 `auto_split` 折行

它的判斷順序:
1. `TALKING` → talking
2. `^轉場` → transition(注意:只判斷繁體!簡體 `转场` 在這裡會掉到 song 分支)
3. `Chaser` → chaser
4. `^(S\d+)[_\s]+(.+)$` → song with 編號
5. 其他 → song without 編號(以 `_` 替換為空格)

> ⚠️ **不對稱**:
> - `parse_song_list_from_text` 接受 `转场`(簡)輸出 `轉場`(繁)
> - `parse_setlist` **只認** `轉場`(繁)
> - 流程上沒問題,因為前者跑過後輸出已經是繁體
> - 但**如果使用者手動在主視窗輸入 `转场`**,會被當 song 處理 → bug 隱形雷
> - **建議**:`parse_setlist` 也加上簡體支援(改 [main.py:168](../../main.py:168) 的 regex)

## 改 parser 的標準流程

1. **先讀本檔案** + [.claude/agents/rd-format-tester.md](../../agents/rd-format-tester.md) + [.claude/skills/testing/SKILL.md](../testing/SKILL.md)
2. **先寫測試**(在 `tests/test_parser.py` 加新 case)— 測試應該**會失敗**(因為新行為還沒實作)
3. 改 parser
4. **跑既有所有測試,全部要通過**
5. 跑新測試,該通過
6. PR 開出來,在描述附「改了 X 行為,影響的測試案例:Y、Z」
7. Reviewer 跑 `rd-format-tester` agent,確認沒有 regression

**禁止**:
- ❌ 在沒測試的情況下「重構 parser 化簡邏輯」
- ❌ 把舞台關鍵字 inline regex 改成 list 但沒測試
- ❌ 在沒測試的情況下調整 Pass 1/2/3 的判斷順序
